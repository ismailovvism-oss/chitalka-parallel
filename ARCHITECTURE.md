# ARCHITECTURE.md — Библиотека исламских знаний

Этот документ описывает целевую архитектуру платформы, частью которой становится
читалка (`app.js`/`parser.js`). Принцип: **статика сейчас, швы под бэкенд на будущее**
— ничего серверного не строим, но проектируем так, чтобы добавить без переписывания.
Текущий рабочий контекст — в `CLAUDE.md`, формат книг — в `SPEC.md`.

## Видение

Не просто читалка, а **база знаний**: библиотека книг с категоризацией, тегами,
фасетами, полнотекстовым поиском и страницами авторов. Читалка — это **слой просмотра**
одной книги; вокруг неё надстраиваются каталог, классификация и поиск. Единый
**формат Контракта** (секторы `sNNN`, пер-язычные сноски) — общий формат всего контента
и главный актив платформы.

## Соответствие текущей реализации (на 14.06.2026)

Уже сделано (бренд **«Библиотека Алькасави»**), и архитектура это учитывает:
- **Полка-библиотека**: книги обложками на «полках», категории-чипсы по полю `tags` в
  `books/index.json`, обложки (`cover` или генерированные), бейдж «стр. N» у начатых.
- **Карточка книги** (тап по обложке → инфо, автор/аннотация/главы/категории, deep-link
  `?info=id`), читалка как вьюер (`?book=id`).
- **Сканы страниц** (`books/tawfiq/img/`, `hasImages`), маркеры страниц в конвейере.
- **Режим самопроверки**, «Сообщить об ошибке» (mailto), «О приложении».
- **Паритет 37/37** по «Тауфик аль-Маннан», Контракт ✓.

Это **фаза 1 каталога**. Дальше — углубление: дерево дисциплин и фасеты поверх плоских
`tags`, полнотекстовый поиск, страницы авторов, вынос контента при росте (см. ниже).

## Один репозиторий или новый?

**Приложение (читалка + библиотека) — один репозиторий, не плодим новые.** Делить
библиотеку и читалку не нужно: это разные виды одного SPA. Разделять стоит **контент и
приложение**, и это уже учтено архитектурно:
- **Источник истины контента** — отдельные авторские репозитории на книгу (как
  `tawfiq-al-mannan`). Они уже отдельные. Так и остаётся.
- **Опубликованный контент** сейчас завендорен в `books/` приложения — нормально для
  десятков–сотен текстовых книг. Выносить в объектное хранилище нужно только при росте
  (тысячи книг или тяжёлые сканы) — тогда `book.json.base`/`media` указывают на CDN, код
  не меняется. **Сейчас новый репозиторий не нужен.**

## Хостинг и Netlify (ответ на частый вопрос)

Приложение статическое — **с Netlify откроется и три уже добавленные книги читаются**
без изменений: деплоим репозиторий, publish-каталог = корень (сборка не нужна, vanilla
JS), `books/` отдаётся как есть, реестр `books/index.json` подхватывается. PWA/SW
работает по HTTPS. Маршрутизация на query-параметрах (`?book=`, `?info=`) — SPA-редиректы
не нужны; если перейдём на path-маршруты (`/book/:id`), добавить `_redirects` с фолбэком
на `index.html`.

Нюансы: Netlify тянет приложение и текущий контент легко (текст крошечный). Сканы в git
(`img/*.jpg`) — ранний сигнал «пора выносить медиа»: на тысячах страниц/книг переводим
сканы и контент в объектное хранилище + CDN. Для масштаба **Cloudflare Pages + R2** —
опция получше (бесплатный egress), но **для старта Netlify ок**.

## Принципы

1. **Static-first, hybrid-ready.** Каталог и поиск генерируются при публикации в
   статические файлы. Каждый такой артефакт спрятан за интерфейсом-провайдером, чтобы
   позже заменить статику на API без правок UI.
2. **Контент отделён от приложения.** Книги не живут в репозитории приложения. Источник
   истины — авторские репозитории (как `tawfiq-al-mannan`); публикуются на CDN/в
   объектное хранилище.
3. **Стабильные id везде** (`bookId`, `sNNN`, `fnN`) — основа глубоких ссылок,
   аннотаций и будущей синхронизации между устройствами.
4. **Контракт — единственный формат тела.** Любой источник приводится к нему адаптером.

## Слои

```
L1  Контент      книги в формате Контракта + медиа (сканы/PDF). Источник истины —
                 авторские репозитории; публикация → CDN / объектное хранилище (R2).
L2  Каталог      book.json (расширенный) + сгенерированные catalog.json, taxonomy.json,
                 authors.json. За провайдером catalog (сейчас статич. JSON → потом API).
L3  Поиск        (а) по каталогу — клиентский, мгновенный (фасеты/теги/автор);
                 (б) полнотекст по содержимому — Pagefind (статически) сейчас →
                     Meilisearch/Typesense (арабская морфология, ранжирование) потом.
L4  Просмотр     читалка как модуль-вьюер: получает bookId → грузит секторы и медиа.
L5  (заложено)   пользователи: синхронизация прогресса/закладок/выделений, аккаунты,
                 пользовательский вклад. Требует бэкенда — сейчас НЕ строим, но швы есть.
```

Приложение — один SPA, разные маршруты: `/` — каталог-браузер, `/book/<id>` — вьюер,
`/author/<id>`, `/search`. Библиотека и читалка — разные виды одной оболочки.

## Хранение

- **Текст книг** (Контракт) — статические файлы. До сотен книг — репозиторий(и)
  контента; на тысячах — объектное хранилище + CDN (репозиторий приложения чистый).
- **Сканы/PDF** — тяжёлое, **только объектное хранилище + CDN** (Cloudflare R2 —
  бесплатный egress; B2/S3 — альтернативы). Никогда не в git и не на статик-хост.
  Манифест книги ссылается на медиа по базовому URL + паттерну (см. `media` ниже).
- **Хостинг приложения** — статик (Cloudflare Pages предпочтительнее Netlify под этот
  масштаб; Netlify ок для старта). Лимит — не место/трафик (текст крошечный), а число
  файлов и раздувание git, поэтому контент вынесен.
- **catalog/индексы** — генерируются при публикации, лежат рядом с приложением;
  шардируются при росте.

## Модель данных

### `book.json` (расширенный манифест)
```jsonc
{
  "bookId": "tawfiq",
  "title": { "ar": "توفيق المنان", "ru": "Тауфик аль-Маннан" },
  "authors": ["ibn-...", ...],     // ссылки в authors.json
  "translators": ["..."],
  "origLang": "ar",
  "languages": ["ar", "ru"],        // первый — оригинал/источник страниц
  "rtl": ["ar"],
  "discipline": "aqida",            // путь в дереве дисциплин (taxonomy.json)
  "tags": ["tawhid", "apologetics"],// многие-ко-многим
  "era": "contemporary",            // или век: "8h"
  "madhhab": null,
  "type": "matn",                   // matn | sharh | taliq | hashiya | ...
  "status": "draft",                // draft | review | verified
  "cover": "cover.webp",
  "description": { "ru": "…" },
  "media": {                        // опционально: сканы/PDF
    "scans": { "base": "https://cdn/…/tawfiq/scan/", "pattern": "p{page}.webp" },
    "pdf":   "https://cdn/…/tawfiq.pdf"
  },
  "counts": { "chapters": 37, "sectors": 1850 },
  "chapters": [ { "file": "01.md", "title": { … } } ]
}
```

### Сгенерированные артефакты (build-шаг)
- `catalog.json` — укороченные записи всех книг для листинга + предрассчитанные
  **фасеты** (язык, дисциплина, автор, эпоха, статус, есть-перевод). Шардируется при росте.
- `taxonomy.json` — дерево дисциплин + контролируемый словарь тегов (с переводами).
- `authors.json` — сущности авторов/переводчиков (имя по языкам, годы, регион) → страницы автора.
- поисковый индекс (Pagefind) — по тексту секторов.

## Поиск

- **По каталогу** — клиентский фильтр по `catalog.json` (мгновенно, без сети).
- **Полнотекст по содержимому** — сейчас **Pagefind**: индекс генерится при публикации,
  грузится шардами лениво, тянет тысячи страниц без сервера. Эскалация — **Meilisearch/
  Typesense** при нужде в арабской морфологии/ранжировании/опечатках (за тем же
  интерфейсом поиска в UI).
- **Арабская нормализация** (обе ветки): снятие огласовок (ташкиль), нормализация
  алифа/хамзы/та-марбуты/йа — иначе поиск по арабскому бесполезен. В читалке уже есть
  `normalizeWithMap` для подсветки — переиспользовать/расширить.

## Конвейер публикации

```
авторские репозитории ─адаптер→ формат Контракта ─validate→ build ─→ publish (CDN/R2)
   (tawfiq и др.)        adapters/        tools/validate.js   catalog.json
                                                              taxonomy/authors
                                                              Pagefind-индекс
```

- `tools/import-tam.js` — **первый адаптер**. Обобщить в `adapters/<источник>`
  (каждый формат-источник → Контракт + манифест).
- `tools/build-catalog.js` (новый) — собрать `catalog.json`/фасеты из всех `book.json`.
- Поисковый индекс генерится тем же шагом.

## Швы под будущий бэкенд (заложить сейчас)

Чтобы L5 добавился без переписывания:
- **Состояние за абстракцией.** Прогресс/закладки/выделения уже в `localStorage` —
  обернуть в `store` с интерфейсом `get/set/list`; позже подменить на sync-адаптер
  (бэкенд). Схему привязать к `bookId` + `sNNN` и версионировать.
- **Каталог за провайдером.** UI ходит в `catalogProvider` (сейчас читает статический
  JSON) — позже тот же интерфейс отвечает из API.
- **Поиск за интерфейсом.** `search(query, filters)` — реализация Pagefind сейчас,
  движок потом.
- **Стабильные id** (есть) — готовы под глубокие ссылки и серверные аннотации.
- **Вклад** — уже git-репозиторий-на-книгу (модель tawfiq); масштабируется в
  редакторский workflow (PR/ветки) без смены формата.

## Дорожная карта

1. **Каталог-MVP (статика).** Расширить `book.json`; `tools/build-catalog.js`;
   вид «библиотека» с фасетами/тегами/деревом дисциплин. Читалка — как есть, вьюером.
2. **Поиск.** Pagefind по содержимому + арабская нормализация; каталожный фильтр.
3. **Медиа.** Сканы/PDF в R2; вьюер сканов в читалке (кнопка «скан» уже заложена).
4. **Авторы/связность.** `authors.json`, страницы авторов, связи книг.
5. **Бэкенд (по потребности).** Поисковый движок; затем аккаунты и синхронизация.

## Развитие читалки (слой L4)

Мультиперевод 3+ языков (ядро готово), глубокие ссылки на сектор, вьюер сканов,
аннотации с привязкой к `sNNN`, оглавление/указатели. Детали — в `BACKLOG.md`.
